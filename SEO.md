# Chaput Organic Discovery Guide

This document records the landing site's English and Turkish organic-discovery architecture. It covers technical SEO, answer-engine readiness, content quality, measurement and release checks. It does not replace product, safety or legal review.

## Positioning

Chaput is a conversation-first social app for iPhone and Android. People discover profiles, connect through approved follow relationships when required, and leave one intentional first Chaput. A reply can turn that opening into a conversation.

The public site must consistently communicate these facts:

- Profiles are private by default.
- A sender leaves one initial Chaput until the receiver replies.
- The normal reply window is 24 hours before archive.
- Revive may be available for eligible archived conversations.
- Conversations are visible only when the applicable privacy rules allow it.
- Whisper is participant-only.
- Reporting, blocking and restriction controls are available.
- Chaput is for people aged 15 and older, subject to a higher local minimum age.
- Chaput is not a random roulette chat, anonymous matching service, dating service or relationship guarantee.

The product idea is summarized as **“Conversations become content.”** This means conversation can become part of profile-led discovery when visibility allows. It never means that a visible conversation is automatically licensed for copying or republication.

## Intent Consolidation

Closely related phrases share one canonical destination when the underlying user need is the same. This keeps the site useful instead of creating near-duplicate keyword pages:

- Faceless, voice-over and no-camera creator questions consolidate under `/for/faceless-creators/`.
- Reaction, commentary and conversation-analysis questions consolidate under `/for/commentary-creators/`.
- Natural-dialogue observation and writing research consolidate under `/for/writers/`.
- Funny-dialogue discovery, dialogue-page planning and podcast-topic planning have separate guides because each requires a materially different workflow and answer.
- Goth community pages split by genuinely different discovery or safety needs; they must not be expanded into stereotype-based or fetishizing keyword variants.

## On-Page Requirements

Every generated content page includes:

- One descriptive H1 and answer-first introduction.
- Unique title, description and canonical URL.
- Reciprocal English/Turkish `hreflang` plus `x-default`.
- Static HTML content in the first response.
- Organization, WebSite, WebPage, MobileApplication, BreadcrumbList and FAQPage structured data.
- Article structured data only for guide content.
- Visible FAQ text that exactly matches FAQ schema.
- Contextual internal links and a single clear store CTA.
- Equal-size official App Store and Google Play badges.
- Explicit light and dark theme contrast.

Do not add review, rating, user-count, price or availability claims without a current verified source. Do not add structured data for content that is not visible on the page.

## Technical Files

- `robots.txt` allows public crawling, declares the sitemap and excludes user/deep-link/account routes from search crawling.
- `sitemap.xml` contains canonical public content and preserves existing legal URLs.
- `llms.txt` provides a concise product definition and canonical sources for AI systems. It supplements normal crawlable pages; it is not a ranking mechanism.
- `feed.xml` exposes selected product and guide updates as Atom entries.
- `scripts/build-content-pages.mjs` renders all EN/TR content pages.
- `scripts/build-sitemap.mjs` rebuilds the sitemap while preserving existing legal URL blocks.
- `scripts/seo-smoke-test.mjs` checks metadata, schema, links, screenshots, sitemap coverage, unsupported claims and critical deep-link/store literals.
- `scripts/submit-indexnow.mjs` optionally submits canonical URLs after deployment.

Run before every release:

```bash
node scripts/build-content-pages.mjs
node scripts/build-sitemap.mjs
node scripts/seo-smoke-test.mjs
```

Only update `lastmod` when the corresponding public content changed materially. Do not change every timestamp solely because a build ran.

## Measurement

Generated content pages reuse the existing anonymous landing analytics endpoints and add an `event` query parameter. The current vocabulary is:

| Event | Trigger |
| --- | --- |
| `organic_landing_view` | Product or discovery content page view |
| `guide_page_view` | Guide page view |
| `use_case_page_view` | Creator use-case page view |
| `goth_hub_view` | Goth hub view |
| `goth_safety_view` | Goth safety page view |
| `faq_open` | A visitor expands a FAQ answer |
| `internal_content_click` | A related-content card is selected |
| `creator_cta_click` | A creator-page hero CTA is selected |
| `goth_community_cta_click` | A goth-page hero CTA is selected |
| `download_ios_click` | App Store badge selection |
| `download_android_click` | Google Play badge selection |
| `legal_link_click` | Footer legal-link selection |

Attribution accepts explicit `source` or `utm_source`, then referrer host. Instagram, TikTok and YouTube are normalized. A locally stored random visitor ID prevents repeated clicks from being counted as new visitors without collecting an email or profile identifier.

The API may initially ignore unknown `event` parameters while preserving its existing visit and store counters. Add server-side event aggregation separately if event-level reporting is required in the admin panel.

## AI and Answer-Engine Discovery

AI discovery depends on the same fundamentals as search discovery: useful crawlable pages, clear factual answers, consistent entities, descriptive internal links and reputable external mentions. There is no separate guaranteed “AI ranking” switch.

- Keep `OAI-SearchBot` able to crawl public pages if Chaput should appear in ChatGPT search citations.
- Manage `GPTBot` separately if training preferences differ from search visibility.
- Monitor ChatGPT referrals using `utm_source=chatgpt.com` where available.
- Keep product facts consistent across the site, App Store, Google Play, press profiles and social profiles.
- Prefer first-party research, product updates, screenshots and transparent safety information over mass-produced keyword pages.

## External Discovery Plan

On-site work cannot manufacture authority. Useful next steps are:

1. Publish accurate Chaput profiles on App Store, Google Play, Product Hunt and relevant company directories.
2. Pitch distinctive product stories to social-app, creator-economy and mobile-product publications.
3. Create original editorial pieces using aggregated, consent-safe observations about conversation openings and dialogue patterns.
4. Collaborate with dialogue, commentary, writing and alternative-community creators without scripting endorsements.
5. Keep publisher name, app name, store URLs and support contact consistent in every profile.
6. Request corrections when third-party descriptions incorrectly call Chaput a dating or random-chat app.

Do not buy links, automate forum promotion, fabricate reviews or publish near-duplicate guest posts.

## Google, Bing and IndexNow Release Checklist

After the deployment is live:

1. Verify `https://chaput.app/robots.txt`, `/sitemap.xml`, `/llms.txt` and representative EN/TR pages return `200`.
2. In Google Search Console, submit `https://chaput.app/sitemap.xml` under **Sitemaps**.
3. Use **URL Inspection** to request indexing for the homepage and the highest-priority hubs: `/what-is-chaput/`, `/use-cases/`, `/guides/`, `/goth/` and their Turkish equivalents. Do not manually request every page at once.
4. In Bing Webmaster Tools, import or verify the site and submit the same sitemap.
5. For IndexNow, generate a private key, publish it as `/<key>.txt`, then run `INDEXNOW_KEY=<key> node scripts/submit-indexnow.mjs` after deployment.
6. Review Search Console coverage, enhancements and Core Web Vitals after Google has recrawled the site. Discovery and indexing are not immediate or guaranteed.
7. Compare organic landing pages, store clicks and source attribution in analytics. Improve pages based on genuine user intent, not impression volume alone.

## Future Localizations

German is the next planned localization. Add it only after every page has human-reviewed German copy. Then generate `/de/` routes, add reciprocal `de` hreflang to all language variants, update sitemap entries and extend smoke tests. Do not publish machine-translated placeholders.

## Protected Areas

Deep-link scripts, app association files, store redirects and all legal/KVKK/contracts are separate release surfaces. SEO changes must not alter them. The smoke test deliberately checks the mobile package, store URLs and public deep-link route pattern for regressions.
